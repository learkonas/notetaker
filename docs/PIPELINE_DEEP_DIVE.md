# Pipeline Deep Dive

This document explains, step by step, how the email inbox -> draft -> Obsidian pipeline works today, with extra detail on summarization, quality/style scoring, related-note retrieval, and style learning. Emails arrive at `notetaker@leonasskau.com`, handled by the self-hosted Cloudflare `email_inbox` worker at `https://inbox.leonasskau.com`.

## High-level architecture

The system runs in two stages:

1. `apps/cloud_job` (cloud-side ingestion and draft generation)
2. `apps/local_sync` (local vault-aware enrichment and note writing)

Data handoff happens through GCS objects under `drafts/YYYY-MM-DD/<messageId>.json` (plus a Markdown preview).

## Stage 1: Cloud pipeline (`apps/cloud_job`)

Execution order in `runPipeline(...)`:

1. `inbox_list`
2. `inbox_get`
3. `parse_email`
4. `extract_links`
5. `summarize`
6. `render_draft`
7. `gcs_put`
8. `inbox_mark_processed`

Important behavior: each skill runs sequentially and receives the previous skill output. If any skill throws, the run fails and later skills do not run.

### 1) `inbox_list`

- Lists emails in the `inbox` folder of the configured mailbox via `GET /api/v1/mailboxes/:mailbox/emails?folder=inbox`.
- Authenticates with Cloudflare Access service token headers.
- Sorted oldest first, capped by `MAX_EMAILS_PER_RUN`.

### 2) `inbox_get`

- Fetches each email in full via `GET /api/v1/mailboxes/:mailbox/emails/:id`.
- Returns full message bodies + metadata.

### 3) `parse_email`

- The inbox worker stores a single `body` field (HTML when available, otherwise plain text).
- Detects HTML bodies and strips them to plain text for summarization, keeping raw HTML for link extraction.
- Emits normalized `ParsedEmail` with:
  - `messageId`, `threadId`, `internalDate` (ISO)
  - `from`, `subject`, `emailText`
  - optional raw `html`

### 4) `extract_links`

- Pulls URLs from both plain text and HTML anchors.
- Normalizes URLs by removing common UTM params.
- Drops known tracking/unsubscribe domains.
- Deduplicates by normalized URL.

### 5) `summarize` (detailed)

This step produces the core draft content (`summary`, `keyPoints`, `analysis`, `questions`, `tags`, evidence, confidence).

#### Default path (`LLM_PROVIDER=mock`)

For each email:

- **Summary**
  - Uses the first sentence of cleaned `emailText`.
  - If sentence splitting fails, truncates the cleaned text to 300 chars.
- **Key points**
  - Splits email text by newline.
  - Keeps trimmed lines longer than 40 chars.
  - Takes the first 5.
- **Evidence quotes**
  - Uses first 3 key points.
  - Each quote is wrapped in quotes and clipped to 220 chars.
- **Confidence**
  - Formula: `clamp(keyPoints.length / 5, min=0.5, max=1.0)`.
  - So confidence is always between `0.5` and `1.0`.
- **Quality flags**
  - Adds `low-confidence` if confidence `< 0.7`.
- **Other fields**
  - `analysis` is currently a placeholder text.
  - `questions` are currently two fixed prompts.
  - `tags` default to `["inbox", "email", "summary"]` then sanitized.
  - `contentHash` is SHA-256 of full `emailText`.

#### OpenAI path (`LLM_PROVIDER=openai`)

Flow:

1. Build a prompt containing subject, sender, full email text, and hyperlinks.
2. Call `POST https://api.openai.com/v1/responses` with configured model.
3. Ask model to return strict JSON with fields:
   - `summary`
   - `keyPoints`
   - `analysis`
   - `questions`
   - `tags`
   - `evidenceQuotes`
   - `confidence`
   - `qualityFlags`
4. Parse returned JSON.
5. Merge parsed fields onto the draft generated by mock heuristics.

Fallback behavior:

- If no usable text or JSON parse fails, the skill keeps the prebuilt heuristic draft for that message.
- Tag sanitization still applies.

#### Tag sanitization rules

Tags are normalized consistently:

- Remove leading `#`.
- Convert whitespace to `-`.
- Lowercase.
- Keep nested `/` groups.
- Remove non `[a-z0-9_-]` characters in each segment.
- Drop empty tags and numeric-only tags (must contain at least one letter).
- Deduplicate.

### 6) `render_draft`

- Creates a lightweight Markdown preview and object prefix.
- Prefix format: `drafts/<YYYY-MM-DD>/<messageId>`.

### 7) `gcs_put`

- Writes:
  - `<prefix>.json` (full structured draft)
  - `<prefix>.md` (rendered preview)

### 8) `inbox_mark_processed`

- Ensures the processed folder exists (creates `ai-processed` if missing).
- Marks each email read and moves it out of `inbox` into the processed folder.
- This happens only after successful GCS persistence, which prevents silent data loss.

## Stage 2: Local pipeline (`apps/local_sync`)

Execution order in `runPipeline(...)`:

1. `vault_scan`
2. `embed` (currently placeholder)
3. `retrieve_related`
4. `style_profile`
5. `quality_score`
6. `enrich_markdown`
7. `style_rewrite`
8. `style_validate`
9. `write_note`
10. `checkpoint`

Input source: `gcs_list_get` loads all draft JSON files under `drafts/` and skips already-processed `messageId`s based on local checkpoint state.

## Related note retrieval (detailed)

Related-note selection is lexical (token overlap), not embedding-based yet.

### Index build (`vault_scan`)

- Recursively scans all `.md` files in vault path.
- For each note stores:
  - `title` (filename without `.md`)
  - full `body` (frontmatter excluded from body text)
  - extracted tags from both inline hashtags and frontmatter `tags`
  - extracted frontmatter `aliases`

### Candidate scoring (`retrieve_related`)

For each draft:

1. Build source token set from: `subject + summary + analysis`.
2. For each indexed note, build note token set from:
   - note title
   - note aliases
   - first 1000 chars of note body
3. Tokenization:
   - lowercase
   - split on non-alphanumeric
   - keep tokens with length `> 2`
4. Similarity formula:
   - `overlap / min(sourceTokenCount, noteTokenCount)`
5. Keep candidates where score `>= 0.1`.
6. Sort descending by score.
7. Keep top 5.

Each selected note gets a generic rationale string. This is deterministic and fast, but can miss semantic similarity when wording differs.

## Quality scoring (detailed)

Quality is assigned in `quality_score` using deterministic heuristics, each clamped to `[1, 5]`.

For each draft:

- `evidenceCoverage = evidenceQuotes.length / max(1, keyPoints.length)`
- `accuracy = 4.5 if evidenceCoverage >= 0.8 else 3.5`
- `coverage = 4 if keyPoints.length >= 3 else 3`
- `insight = 4 if analysis.length > 80 else 3`
- `clarity = 4 if summary.length > 40 else 3`
- `actionability = 4 if questions.length >= 2 else 3`
- `overall = average of the 5 dimensions`

Review gate:

- `needsReview = (accuracy < 4) OR (overall < 4)` (plus any existing needsReview flag)
- If gate fails, reason `"Quality gate did not pass"` is appended.

Implication today: because values are mostly 3/4/4.5 buckets, quality is easy to reason about but coarse.

## Style learning and style score (detailed)

Style behavior is split across profile -> rewrite -> validate.

### 1) Learn style profile (`style_profile`)

Source notes:

- Uses `STYLE_SAMPLE_PATH` (default `Example notes`).
- If that path is not found from current cwd, tries a repo-relative fallback.

Profile extraction:

- Reads all sample `.md` files.
- Extracts `## Section Title` headings from each file.
- Creates `requiredSections` as unique section titles (first 8 only).
- Computes:
  - `avgLength` (mean character length)
  - `avgBullets` (mean `- ` bullet count)
  - `preferredTagPrefix` (`#`)

Writes profile to `shared/style/style_profile.json` and stores it in runtime context.

Important: currently only `requiredSections` is enforced downstream. `avgLength` and `avgBullets` are recorded but not yet used by scoring/rewrite.

### 2) Build note body (`enrich_markdown`)

Creates final Markdown with frontmatter and fixed core sections:

- `# <subject>`
- `## TL;DR`
- `## Key Ideas`
- `## Analysis`
- `## Questions`
- `## Source Links`
- `## Related Notes`

Frontmatter also includes:

- `quality_score`
- `style_score`
- `needs_review`
- related note titles and pipeline metadata

### 3) Force missing learned sections (`style_rewrite`)

For each section in learned `requiredSections`:

- If the generated Markdown does not include `## <section>`, append:
  - `## <section>`
  - `- TODO: section added by style rewrite`

This guarantees section presence before validation, but placeholders may reduce editorial quality.

### 4) Assign style score (`style_validate`)

Given required sections and final Markdown:

- `missing = requiredSections not present`
- `styleScore = 1.0` when no missing sections
- else `styleScore = max(0, 1 - 0.15 * missing.length)`

Style review gate:

- If `styleScore < 0.8`, set `needsReview = true`
- Append reason `"Style score below threshold"`

Because `style_rewrite` runs before `style_validate`, style score is often pushed up by auto-added TODO sections.

## Final routing and persistence

`write_note` routes by gate result:

- `needsReview=true` -> review folder (default `Inbox Notes/Needs Review`)
- else -> inbox folder (default `Inbox Notes`)

Filename format:

- `<YYYY-MM-DD> - <slugified-subject>.md`
- Adds numeric suffix (`-2`, `-3`, ...) on collision.

Finally `checkpoint` records processed message IDs in `.local/checkpoint.json` so reruns do not duplicate notes.

## Current strengths and limits

Strengths:

- Fully deterministic fallback path.
- Explicit quality/style gates with reasons.
- Idempotency protections on both cloud and local sides.
- Simple, inspectable logic.

Current limits:

- Quality/style are mostly heuristic buckets, not model-evaluated rubrics.
- Related-note matching is lexical overlap only.
- Style rewrite inserts TODO placeholders rather than semantic rewrites.
- `avgLength` and `avgBullets` are computed but not enforced yet.
